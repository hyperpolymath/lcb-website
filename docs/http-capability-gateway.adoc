= HTTP Capability Gateway for LCB defense

Capability enforcement for LCB now starts at the origin rather than the CDN edge. The authoritative implementation lives in `wp-content/mu-plugins/origin-governance-gateway.php`, which checks capability tokens on state-changing API requests and only treats the Cloudflare `http-capability-gateway` worker as an optional `/api/*` prefilter.

== Why we use it
* Governs sensitive verbs (DELETE/PUT/PATCH) on `wp-json`, `api`, `micropub`, and `xmlrpc` surfaces so external automation cannot trigger writes without explicit authority.
* Lets first-party WordPress sessions keep using normal WordPress auth while requiring external automations to present `X-Capability-Token`.
* Keeps the final decision close to the application, so Cloudflare drift or bypasses do not remove the control.

== Integration points
1. Canonical enforcement: `wp-content/mu-plugins/origin-governance-gateway.php` runs inside WordPress and validates HS256 JWT capability tokens using `SINOPLE_CAPABILITY_SECRET`.
2. Origin fast-fail: Varnish/OpenLiteSpeed can block obvious bad requests earlier, but those layers should mirror the origin rules rather than invent a separate policy.
3. Optional internet edge: the Cloudflare `http-capability-gateway` worker can reject unauthorised `/api/*` traffic early when `enable_capability_gate=true`, but it must be treated as a helper, not the only line of defence.
4. Token claims should stay aligned with the published AIBDP policy, the expected API surfaces, and any future Svalinn/Vörðr audit expectations.

== Next actions
* Provision `SINOPLE_CAPABILITY_SECRET` in the runtime environment and test signed tokens against the protected API surfaces.
* Keep the MU-plugin capability mapping and any optional Cloudflare worker routes aligned whenever a new API namespace is introduced.
* Tie the origin enforcement logs into the `feedback-o-tron` pipeline so denied capability checks are immediately reported.
