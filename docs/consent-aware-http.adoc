= Consent-aware HTTP for LCB

The LCB site follows the consent-aware HTTP (AIBDP) model so that bots and AI agents must declare their intent before interacting with the hardened runtime. The canonical enforcement point is the WordPress origin itself: `wp-content/mu-plugins/origin-governance-gateway.php` rejects automated requests that do not present valid consent, returning HTTP 430 (Consent Required) and linking callers to `/.well-known/aibdp.json`.

== Policy sources
- Follow https://consent-aware-http.org for the specification, schema versions (e.g., `schemas/aibdp-v0.2.json`), and status codes. We keep one copy of the schema locally in `docs/consent-aware-http-schema-v0.2.json` (mirror from the canonical repo) so the Rust agent can validate incoming declarations.
- The site publishes its own `.well-known/aibdp.json` describing what consent fields it accepts (purpose, data types, credential level). Update this file whenever the site’s data collection strategy changes.
- Keep the MU-plugin enforcement logic, `services/varnish/consent-check.vcl`, and any Cloudflare worker rules aligned with that published policy.

== Runtime behavior
1. `/.well-known/*` remains directly reachable so automated clients can discover the consent policy before attempting protected requests.
2. Requests identified as automated, bot-like, or explicitly AIBDP/capability-aware must present `X-AIBDP-Consent: accepted`; otherwise the origin returns HTTP 430 plus `Preference-Required: consent-aware-http`.
3. Accepted consent must also include `X-AIBDP-Purpose`, and the origin currently allows only `research`, `training-excluded`, `monitoring`, and `security-audit`.
4. Varnish, OpenLiteSpeed, or Cloudflare may reject the same request earlier for efficiency, but those layers are treated as fast-fail helpers rather than the source of truth.

== Enforcement
- Canonical: `wp-content/mu-plugins/origin-governance-gateway.php` inspects requests before theme/plugin routing and emits the authoritative JSON denial response.
- Fast-fail at origin edge: `services/varnish/consent-check.vcl` can synthesize the same 430/403 response before WordPress executes.
- Optional internet edge: Cloudflare `consent-aware-http` may be enabled to reject obvious violations earlier, but it must mirror the origin policy instead of replacing it.
- Whenever a new bot interface is added (API, GraphQL, CLI), update `.well-known/aibdp.json`, the MU-plugin filters/exemptions, and this doc together.

== Monitoring & auditing
- Log MU-plugin denials and Varnish rejects alongside Vörðr traces so the next audit can show which requests were denied and why.
- Share the same policy text in the `cadre-router` UI and operator docs so maintainers can see the enforced consent posture without inspecting server config.
